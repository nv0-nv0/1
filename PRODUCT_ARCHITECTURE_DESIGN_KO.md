# 제품 품질·성능 정밀 설계서

이 문서는 4개 제품을 기능 나열형이 아니라 **입력 기준 → 처리 엔진 → 결과 계약 → 품질 게이트 → 성능 설계 기준 → 실패 대응 → 확장 경로** 구조로 재정의한 기준 문서입니다.

주의: 아래 성능 항목은 **실환경 계측 결과가 아니라 설계 기준**입니다. 실운영 수치는 별도 계측으로 확정해야 합니다.

## Veridion

- 제품 정의: 온라인 사업자 사이트에서 놓치기 쉬운 법률 준수 항목을 점검하고, 예상 과태료 미리보기와 맞춤 수정까지 한 번에 정리하는 준법 운영 제품
- 핵심 문제: 사이트는 운영 중이지만 법률 준수 누락 때문에 과태료, 분쟁, 신뢰 저하가 걱정되는 상황
- 설계 요약: 법률 준수 누락을 발견하는 데서 끝나지 않고, 과태료 위험·수정 우선순위·변경 감시까지 한 기록선으로 묶는 준법 운영 구조입니다.

### 입력 기준
- 대표 URL, 업종, 판매/예약 방식, 수집하는 개인정보 항목을 먼저 받습니다.
- 체크아웃·회원가입·문의폼·광고 랜딩처럼 위험이 큰 공개 구간을 우선 범위로 확정합니다.
- 법률 자문이 필요한 영역과 자동 점검이 가능한 영역을 처음부터 분리합니다.

### 엔진 구조
- 페이지 수집 및 공개 구간 분류
- 준수 항목 룰셋 매칭
- 누락·취약 지점 탐지
- 예상 과태료 미리보기 계산
- 맞춤 문구·고지문 발행
- 법령 변경 감시 및 재점검 후보 생성

### 결과 계약
- 위험도 기준 스캔 리포트
- 예상 과태료 미리보기 표
- 페이지별 수정 우선순위
- 약관·고지·배너 수정안
- 변경 감시용 재점검 큐

### 품질 게이트
- 법률 범주와 페이지 위치가 맞지 않으면 수정안 발행 전에 보류합니다.
- 예상 과태료는 확정 금액이 아니라 범위형 미리보기로만 제시합니다.
- 자동 생성 문구는 적용 전 검토가 필요하다는 안내를 항상 함께 붙입니다.
- 같은 항목을 중복 경고하지 않도록 규칙 충돌을 먼저 정리합니다.

### 성능 설계 기준
- 첫 스캔에서는 전체 해설보다 우선순위와 리스크가 먼저 보이게 설계합니다.
- 1인 운영 기준에서 URL·업종 중심의 최소 입력으로 시작하도록 합니다.
- 법령 변경이 생기면 전체를 다시 읽기보다 영향 가능성이 높은 페이지부터 재점검합니다.

### 실패 대응
- 크롤링이 막히는 페이지는 수동 URL 추가 입력 경로로 전환합니다.
- 법령 소스가 바뀌거나 API가 실패하면 마지막 정상 규칙셋과 변경 로그를 함께 보관합니다.
- 판단이 애매한 항목은 자동 수정안 대신 검토 필요 항목으로 내보냅니다.

### 확장 경로
- 업종별 규칙셋 추가
- 멀티 도메인/멀티 스토어 통합 점검
- 법령 API live 연결
- 재점검 예약 발행과 관리자 승인 흐름

## ClearPort

- 제품 정의: 준비 서류 기준, 고객 안내, 보완 요청 문장을 실무에 맞게 다시 정리해 왕복 시간과 설명 피로를 줄이는 문서 제품
- 핵심 문제: 서류 기준과 안내 문장이 자주 달라져 고객 응대와 일정 관리가 불안정한 팀
- 설계 요약: 서류 기준이 사람마다 달라지는 문제를 줄이기 위해, 준비 기준·보완 요청·진행 안내를 같은 문장 체계로 묶는 실무 문서 운영 구조입니다.

### 입력 기준
- 업무 유형, 서류 종류, 고객군, 필수 확인 항목을 먼저 받습니다.
- 기존 안내문과 보완 요청 사례를 있으면 같이 받아 기준 차이를 파악합니다.
- 나라·운송 방식·통관 단계처럼 문장 기준이 달라지는 분기 조건을 먼저 정리합니다.

### 엔진 구조
- 서류 기준 수집
- 중복/충돌 기준 정리
- 누락 점검 항목 구조화
- 보완 요청 문장 템플릿화
- 단계별 고객 안내문 분기 생성
- FAQ/내부 운영 메모 생성

### 결과 계약
- 준비 서류 기준표
- 누락 점검 체크리스트
- 보완 요청 템플릿
- 단계별 고객 안내문
- FAQ 초안 및 내부 운영 기준

### 품질 게이트
- 같은 단계에서 서로 다른 안내가 나오지 않도록 기준표를 먼저 단일화합니다.
- 법적 확정 해석이 필요한 문장은 업무 안내와 구분해 표기합니다.
- 고객이 오해하기 쉬운 책임 범위·기한 문장은 항상 명시형으로 다시 씁니다.
- 반복 문의가 많은 항목은 FAQ에 강제 반영합니다.

### 성능 설계 기준
- 담당자가 바뀌어도 같은 답이 나오도록 문장 구조를 표준화합니다.
- 보완 요청 왕복 횟수를 줄이기 위해 누락 조건과 예시를 함께 보여 줍니다.
- 새 사례가 생기면 템플릿을 통째로 다시 쓰지 않고 분기만 추가하게 설계합니다.

### 실패 대응
- 조건이 불명확한 요청은 기본 안내 대신 추가 확인 필요 상태로 보냅니다.
- 나라/품목별 예외가 많으면 공통 템플릿과 예외 메모를 분리합니다.
- 내부 기준 충돌이 발견되면 최신본 잠금과 변경 이력을 함께 남깁니다.

### 확장 경로
- 품목별 체크리스트
- 언어별 고객 안내
- CRM 연동 발송 템플릿
- 반복 문의 자동 응답 보조

## GrantOps

- 제품 정의: 지원사업 공고를 읽는 데서 멈추지 않고, 제출 직전까지 필요한 준비 구조와 역할 분담을 잡아주는 실행형 제품
- 핵심 문제: 공고는 찾았지만 제출 직전에 자료, 일정, 역할이 한꺼번에 엉키는 팀
- 설계 요약: 공고를 읽고 끝나는 것이 아니라, 제출 직전까지 필요한 준비와 역할을 끊기지 않게 연결하는 제출 운영 구조입니다.

### 입력 기준
- 공고 원문, 마감일, 제출 채널, 준비 중인 자료를 먼저 받습니다.
- 필수 제출물과 선택 제출물을 구분해 리스크 수준을 나눕니다.
- 담당자, 승인권자, 외부 협력자처럼 역할 분기 정보를 초기에 고정합니다.

### 엔진 구조
- 공고 핵심 해석
- 제출 항목 구조화
- 일정 역산
- 역할 분담 매핑
- 누락 리스크 탐지
- 보완 대응 문장과 제출 체크리스트 생성

### 결과 계약
- 공고 요약본
- 제출 체크리스트
- 역산 일정표
- 역할 분담표
- 누락 리스크 메모
- 보완 요청 대응 문장

### 품질 게이트
- 필수 제출물과 참고 자료를 혼동하지 않게 우선순위를 분리합니다.
- 마감 직전 병목이 큰 승인 단계는 일정표에서 별도 표시합니다.
- 근거가 없는 해석은 공고 원문 링크와 함께 검토 필요로 남깁니다.
- 제출 직전 체크리스트는 담당자별 확인 지점을 명시합니다.

### 성능 설계 기준
- 공고를 처음 읽는 시간보다 제출 직전 재작업 시간을 줄이는 데 초점을 둡니다.
- 역할이 겹치는 구간을 먼저 드러내서 커뮤니케이션 비용을 낮춥니다.
- 다음 공고에도 재사용할 수 있게 체크리스트와 역할 표를 자산화합니다.

### 실패 대응
- 공고 내용이 애매하면 확정 안내 대신 해석 후보와 확인 질문을 분리합니다.
- 마감 변경이나 추가 공지가 나오면 기존 일정표를 덮지 않고 버전 비교를 남깁니다.
- 필수 자료 누락이 감지되면 제출 순서보다 누락 복구를 먼저 띄웁니다.

### 확장 경로
- 기관별 공고 템플릿
- 반복 제출용 템플릿 팩
- 알림 캘린더 연동
- 협업 승인 기록 연동

## DraftForge

- 제품 정의: 초안은 빠르게 나오지만 검토, 승인, 최종본 정리에서 시간이 새는 팀을 위해 마지막 단계를 더 빠르고 정확하게 정리하는 콘텐츠 운영 제품
- 핵심 문제: 초안은 금방 나오지만 승인과 최종본 정리에서 일정이 늦어지는 팀
- 설계 요약: 초안을 빨리 만드는 것보다, 검토·승인·최종본 정리를 더 안정적으로 끝내는 데 초점을 둔 최종화 운영 구조입니다.

### 입력 기준
- 초안, 채널, 승인자, 마감일, 필수 반영 포인트를 먼저 받습니다.
- 버전이 여러 개인 경우 최신본 기준과 비교 기준을 먼저 고정합니다.
- 게시 채널별 형식 차이가 있으면 본문·썸네일·CTA 기준을 분리합니다.

### 엔진 구조
- 초안 수집
- 검토 기준 정리
- 의견 충돌 정렬
- 채널별 최종본 분기
- 버전명/파일명 표준화
- 발행 전 체크리스트 생성

### 결과 계약
- 검토 흐름 정리본
- 승인 체크리스트
- 채널별 최종본 비교표
- 버전 관리 규칙
- 발행 전 QA 체크리스트

### 품질 게이트
- 누가 언제 승인하는지 없는 상태에서는 최종본 확정으로 넘기지 않습니다.
- 채널별 형식 제약이 다르면 단일 본문으로 처리하지 않고 분기본을 만듭니다.
- 의견 충돌은 반영/보류/제외 사유로 나눠 기록합니다.
- 최종본 파일명과 게시 버전이 다르면 발행 직전 경고를 띄웁니다.

### 성능 설계 기준
- 초안 이후의 대기 시간을 줄여 전체 마감 속도를 높입니다.
- 버전명·체크리스트를 표준화해 같은 실수를 반복하지 않게 합니다.
- 콘텐츠 수가 늘어도 최종본 정리 비용이 선형으로 커지지 않게 설계합니다.

### 실패 대응
- 승인자가 응답하지 않으면 대기 상태와 대체 경로를 분리 표시합니다.
- 최신본이 불명확하면 자동 최종화 대신 비교 상태로 멈춥니다.
- 채널별 누락이 생기면 공통 완료 처리 대신 부분 완료로 기록합니다.

### 확장 경로
- 채널별 발행 템플릿
- 승인 로그 자동화
- 콘텐츠 캘린더 연동
- 재발행/리패키징 운영
