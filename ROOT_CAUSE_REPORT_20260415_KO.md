# NV0 원인분석 보고서

## 1. 확인된 뿌리 원인
1. 제품 상세 핵심 정보가 정적 HTML에 충분히 박혀 있지 않아, JS 로딩 실패·캐시 불일치·배포 중간 상태에서 빈 화면처럼 보일 수 있었습니다.
2. 회사/정책 페이지의 고지 밀도가 낮아 신뢰와 법적 안내가 약하게 보였습니다.
3. 로컬 검증 환경에서 Veridion 스캔이 내부 주소를 기본 차단해, 개발·스모크 테스트 환경에서는 기능 불능처럼 보일 수 있었습니다.
4. `.env.local`과 `.env.example`가 즉시 실행형 로컬 기준보다 운영용 placeholder에 가까워, 그대로 실행하면 관리자 토큰/호스트/원본 설정 때문에 검증 동선이 꼬일 수 있었습니다.
5. post-build override가 핵심 페이지를 적극적으로 덮어쓰지 않아, 개선된 SSR 설계가 실제 dist 결과물에 반영되지 않았습니다.

## 2. 실제 조치
- 제품 상세를 SSR 중심 구조로 강제 반영하도록 override 범위를 확장했습니다.
- 회사/제품/문제별 시작/공통 엔진/게시판 허브를 개선본으로 dist에 재출력하도록 수정했습니다.
- 개인정보처리방침/환불정책/쿠키 안내를 더 구체적인 고지형 문서로 보강했습니다.
- Veridion 로컬 회귀를 위해 localhost 기반 개발 환경에서 내부 주소 스캔을 허용하도록 검증 로직을 보강했습니다.
- `.env.local`, `.env.example`를 즉시 실행 가능한 로컬 기준값으로 조정했습니다.
- 감지 가능한 정적 문제를 세는 `site_issue_inventory.py`를 추가하고, 현재 감사 범위 기준 0건까지 정리했습니다.

## 3. 검증 결과
- `smoke_release.py` 통과
- `product_runtime_e2e.py` 통과
- `veridion_runtime_regression.py` 통과
- `site_issue_inventory.py` 기준 감지 가능 문제 0건

## 4. 남는 한계
- 실도메인 `nv0.kr` 반영, 실제 Toss 실결제, 실제 Coolify 반영까지는 이 환경에서 직접 집행할 수 없습니다.
- 사업자 상세 고지 중 대표자명/주소/전화 등 사실값이 패키지 내부 데이터에 없으므로 제가 임의로 생성하지는 않았습니다. 이 정보는 확인되지 않았습니다.
